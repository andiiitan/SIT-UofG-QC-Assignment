# BB84 Quantum Key Distribution Simulation

This repository contains a Python-based simulation of the BB84 quantum cryptography protocol using IBM's Qiskit framework. It explores how quantum mechanics can be leveraged to establish a secure, shared cryptographic key between two parties, and demonstrates the physical impossibility of undetectable eavesdropping on a quantum channel.

## Repository Structure

* **`BB84-Plain.ipynb`**: Simulates the ideal execution of the BB84 protocol over a secure quantum channel. It demonstrates state preparation, transmission, measurement, and the final key sifting process to establish a matching secret key between Alice and Bob.
* **`BB84-Attacker.ipynb`**: Introduces an active eavesdropper ("Eve") utilizing an Intercept-and-Resend attack. It proves how the No-Cloning Theorem forces Eve to collapse the quantum states, ultimately allowing Alice and Bob to detect her presence.
  * *Bonus*: This notebook also includes a secondary demonstration of an advanced **Entanglement (CNOT) Attack**.

## Core Protocol Mechanics

This simulation is built to accurately reflect the physical requirements of quantum cryptography:

* **Hardware-Simulated Quantum Randomness:** Classical pseudo-random number generators (like Python's `random` module) are not used for any cryptographic decisions. All random bits and basis selections are generated by applying a Hadamard gate to initialize a $|+\rangle$ superposition state and measuring the physical outcome. 
* **Agent-Based Architecture:** The simulation executes sequentially to represent the linear flow of time, but is architecturally divided into distinct agent phases (Alice's Preparation, Eve's Interception, Bob's Measurement) to clearly illustrate the localized information each party possesses.
* **Statistical Eavesdropper Detection:** To verify the integrity of the channel, Alice and Bob publicly sacrifice a sample subset of their sifted key. The system actively calculates the error rate in this sample; if the proportion of disrupted qubits exceeds a predefined security threshold (e.g., 10%), the protocol correctly flags the channel as compromised and discards the key.
